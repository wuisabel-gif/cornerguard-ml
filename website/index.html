CornerGuard ML

Predictive vehicle stability warning for race cars.

Most dashboards show what already happened. CornerGuard reads steering, speed, yaw rate and lateral acceleration to estimate whether the car is likely to lose grip in the next 0.5–1.5 seconds if the driver holds the current inputs — flagging risk before instability is obvious.

Motivation

Race cars live at the grip limit. CornerGuard watches the edge.

A race car is driven on purpose right up against the tire's grip ceiling. There, a small change in steering, brake release, throttle, surface or speed can tip into understeer, oversteer, or a track exit within a fraction of a second.

Conventional telemetry

Reports the past

Standard dashboards plot what already occurred — the slide, the lockup, the off. By the time the trace shows it, the driver has already felt it.

CornerGuard

Estimates the next moment

It projects current inputs forward and asks a different question: is this combination of steering, speed and rotation about to exceed what the tires can hold?

Earlier cues for the driver

How it works

From raw telemetry to a forward-looking warning state

A short rolling window of signals feeds a classifier trained to recognize the run-up to instability, not just instability itself.

01 · ingest

Telemetry

Steering, speed, yaw rate and lateral g sampled from the car or simulator.

02 · buffer

2 s rolling window

Recent samples are framed into a fixed window of context the model can read.

03 · infer

Future-risk classifier

The model estimates the probability that grip is lost within the prediction horizon.

04 · output

Stable / warning / critical

A probability and a discrete warning state, ready to log or surface live.

Vehicle model

A bicycle model gives the math a physical anchor

Before any learning, a simple single-track model predicts how fast the car should be rotating for a given steering angle and speed. The gap between expected and measured rotation is a physically meaningful feature.

# expected yaw rate (single-track / bicycle model)
r_expected = (v / L) · tan(δ)
# how far reality departs from the model
yaw_error = r_measured − r_expected

The expected yaw rate is what the geometry predicts for the current speed and steering. The yaw error captures how much the real car disagrees with that prediction — the signal that something at the tires is changing.

v  forward speed
L  wheelbase
δ  front steering angle
r  yaw rate (rotation about vertical axis)
Understeer FRONT WASHES OUT

When |r_measured| < |r_expected|, the car turns less than the steering commands. The front axle is saturating: the nose pushes wide of the intended line.

Oversteer REAR STEPS OUT

When |r_measured| > |r_expected|, the car rotates faster than the geometry predicts. The rear is losing grip and the tail begins to come around.

Sign and magnitude of yaw_error, together with steering rate and lateral load, let the classifier separate a planted car from one drifting toward either failure mode — well before it fully lets go.

ML roadmap

Start simple and interpretable, grow toward sequence models

The baseline is intentionally boring and explainable. Later versions add temporal modeling as the dataset and validation mature.

BaselineVersion 1

Random Forest / XGBoost

Tree ensembles over engineered features from the last 2 seconds of telemetry. Fast, interpretable, and a clear accuracy floor to beat.

▣ tabular window features
Version 2

LSTM / Temporal CNN

Sequence models that read the raw signal window directly, learning temporal patterns the hand-built features miss.

▣ learned temporal structure
Version 3

Transformer sequence

Attention over longer telemetry sequences for richer context and multi-signal prediction across a corner.

▣ long-range attention
Interface

Desktop dashboard

Replay CSV logs, inspect yaw error, and highlight warning zones so engineers can compare drivers and setups.

Project structure & quick start

One repo: embedded firmware plus a Python ML toolkit

Firmware lives alongside the model code, with scripts to generate simulated data, train the baseline, replay inference over a run, and feed a future desktop dashboard.

# repository layout cornerguard/ ├── src/ # embedded firmware ├── include/ # firmware headers ├── cornerguard_ml/ # python ml tools ├── scripts/ │ ├── generate_sim_data.py │ ├── train_baseline.py │ └── replay_inference.py ├── data/processed/ └── models/
bash — cornerguard
# set up an isolated environment$ python3 -m venv .venv$ source .venv/bin/activate$ pip install -e ".[ml,test]" # generate simulated cornering data$ python scripts/generate_sim_data.py --runs 200 \ --out data/processed/simulated_cornering.csv # train the baseline classifier$ python scripts/train_baseline.py \ --data data/processed/simulated_cornering.csv \ --model-out models/cornerguard_baseline.joblib # replay inference over a run$ python scripts/replay_inference.py \ --data data/processed/simulated_cornering.csv \ --model models/cornerguard_baseline.joblib

Safety note

  • CornerGuard is a warning, logging and analysis system — it estimates and reports risk only.
  • In this first version it must not control steering, throttle or braking, and is not wired to any actuator.
  • It is a research prototype and baseline ML system, not a production- or safety-certified product, and accuracy is not guaranteed.
  • Any on-car testing should happen only in safe, controlled environments with appropriate supervision.
